# Executing a wargame

A wargame runbook is generated by `/wargame`; this is how it gets **run**. The
runbook pre-simulated action → reaction → counteraction so a cheaper executor
can walk it confidently. Execution enforces three things the prose alone can't:
the **ledger gate**, **fork-following**, and **abort halts**.

## Invocation

```
/execute-wargame <mission>        # run one mission's wargame
```

`<mission>` matches a file in `.wargame/wargames/<mission>.md` (and its brief in
`.wargame/tasks/<mission>.md`). Run from the repo root that holds `.wargame/`.

## Protocol — do these in order, do not skip

1. **Gate on the ledger first.** Run
   `bash <skill>/scripts/gate.sh <repo>/.wargame`. If it exits non-zero, **STOP**
   — print the unresolved `(variable)` list and ask the user to resolve them (or
   explicitly authorise assumed values). Never begin moves with an unresolved
   placeholder that a move actually reads. This is a hard gate, not advice.

2. **Load context.** Read `tasks/<mission>.md` (the order + brief),
   `wargames/<mission>.md` (the moves), and `assumptions.md` (what's taken as
   given). Restate the mission and the abort conditions in one line before Move 0.

3. **Walk moves in order.** For each `## Move N`:
   - Execute the **Action** exactly as written.
   - Compare what you observe to **Expected observation** (✅ worked / ❌ didn't).
   - If it matches a **Fork** trigger, jump to that route's move — do not fall
     through to N+1 unless the "Else" branch says so.
   - If the **Most-likely failure** signals appear, apply its **Countermove**
     before deciding the fork.
   - Respect **Stop-depth** notes — don't simulate deeper than the runbook chose to.

4. **Check abort conditions before every move.** If the current state matches any
   line in the mission's **Abort conditions** block, **HALT immediately** and
   report which condition fired. Do not work around an abort — that's the point
   of naming it.

5. **Report the walk.** End with the path taken (which forks, which moves
   skipped), where it stopped (done / halted-at-abort / blocked-on-observation),
   and any move whose Expected observation was **wrong in reality** — that's a
   runbook bug: note it so `/wargame` can patch that move.

## Rules

- The gate is non-negotiable. Unresolved ledger row that a move reads → stop and ask.
- Fork triggers are observations, not vibes. If you can't observe the trigger, you can't take the route — say so.
- Abort = halt + report, never work around.
- You are executing, not re-wargaming. If reality diverges from every documented
  branch, stop and report the gap rather than inventing a new plan on the fly.
